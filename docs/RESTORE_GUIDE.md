# Руководство по восстановлению проекта 500

Этот документ нужен для аварийного восстановления проекта.

Цель: даже если проект сломался, потерялся контекст или непонятно, что было изменено, по этому файлу можно восстановить понимание и выбрать безопасный путь отката.

## 1. Главное правило восстановления

Не пытаться сразу чинить всё.

Сначала нужно определить:

1. Что именно сломано.
2. Когда это было рабочим.
3. Какие файлы менялись последними.
4. Есть ли резервная ветка или релиз.
5. Можно ли восстановить только один файл, а не весь проект.

## 2. Что уже есть для отката

Перед наполнением документации Антон вручную сделал:

- резервную ветку;
- релиз/точку отката.

Эту ветку нельзя использовать для обычной разработки. Она нужна только как аварийная копия.

## 3. Порядок восстановления понимания проекта

Если проект открыт после долгого перерыва, читать в таком порядке:

1. `README.md` — краткое описание и запуск.
2. `docs/CHAT_CONTEXT.md` — что обсуждалось в чатах.
3. `docs/PROJECT_OVERVIEW.md` — общая идея игры.
4. `docs/CURRENT_STATE.md` — текущее состояние по файлам.
5. `docs/ARCHITECTURE.md` — архитектура и будущий план.
6. `docs/DATA_STRUCTURE.md` — враги, башни, волны.
7. `docs/BUGS_AND_FIXES.md` — известные баги и риски.
8. `ROADMAP.md` — куда двигаться.
9. `CHANGELOG.md` — что менялось.

## 4. Порядок технической проверки

### Шаг 1. Скачать проект

```bash
git clone https://github.com/sunpole/500.git
cd 500
```

### Шаг 2. Проверить ветку

```bash
git branch
```

Должна быть ветка:

```text
main
```

### Шаг 3. Установить зависимости

```bash
npm install
```

### Шаг 4. Запустить локально

```bash
npm run dev
```

Открыть адрес, который покажет Vite.

### Шаг 5. Проверить сборку

```bash
npm run build
```

Если команда прошла успешно, появится папка:

```text
dist/
```

## 5. Что проверять в браузере

После запуска игры проверить:

- страница открывается;
- canvas виден;
- кнопка запуска следующей волны видна;
- UI-панель видна;
- башни выбираются;
- башни ставятся на поле;
- нельзя ставить башни на вход и выход;
- враги появляются;
- враги идут к базе;
- башни стреляют;
- лазерные башни отображают луч;
- DOT/яд наносит урон;
- деньги начисляются;
- здоровье уменьшается, если враг дошёл;
- волны заканчиваются;
- следующая волна стартует;
- победа/поражение отображаются.

## 6. Что проверять в консоли браузера

Открыть DevTools → Console.

Искать ошибки:

```text
ReferenceError
TypeError
SyntaxError
Failed to fetch
Cannot read properties of undefined
is not defined
```

Каждую ошибку записывать в:

```text
docs/BUGS_AND_FIXES.md
```

## 7. Быстрая диагностика по симптомам

### Белый экран или пустая страница

Проверить:

- есть ли ошибка в консоли;
- загружается ли `js/game.js`;
- нет ли ошибки импорта;
- правильно ли указан `<script type="module" src="js/game.js"></script>`.

### Canvas не найден

В `index.html` должен быть:

```html
<canvas id="game"></canvas>
```

В `game.js` код ищет:

```js
document.getElementById('game')
```

Если id изменён, игра не стартует.

### Не появляются волны

Проверить:

- `waveData` импортируется из `constants.js`;
- `state.wave` начинается с 0;
- `state.waveTimeoutActive` не завис навсегда;
- `updateTimers(dt)` вызывается;
- `launchWave(state.wave)` вызывается;
- нет ошибки в `spawnEnemy`.

### Не появляются враги

Проверить:

- `enemyData[eidx]` существует;
- `state.path` создан;
- `generateEnemyPath()` отработал;
- `spawnEnemy()` не пишет в консоль `Неизвестный враг`;
- в волнах нет индекса врага, которого нет в `enemyData`.

### Башни не ставятся

Проверить:

- выбрана ли башня;
- хватает ли денег;
- клетка свободна;
- `canPlaceTower(x, y)` возвращает true;
- путь врагов не перекрывается;
- нет ошибки с переменными `grid` или `towers`.

### Башни не стреляют

Проверить:

- есть ли враги в радиусе;
- `updateTowersShooting(dt)` вызывается;
- cooldown уменьшается;
- `towerData[t.type]` существует;
- снаряды добавляются в `state.bullets`;
- лазерные башни имеют `laser: true`.

### Деньги не начисляются

Проверить:

- при смерти врага выполняется `state.money += reward`;
- у врага есть `conf.reward`;
- вызывается `updateUI()`.

### Игра зависает

Проверить:

- слишком много врагов в поздних волнах;
- слишком много логов `console.log` в `update()` и `draw()`;
- слишком частые снаряды у башен с маленьким cooldown;
- огромные DOT-стаки.

## 8. Возможные известные технические риски в коде

Эти пункты нужно проверять особенно внимательно.

### Риск 1. Логи в каждом кадре

В `game.js` есть логи в `update()` и `draw()`.

Потенциальная проблема: консоль может быстро переполняться, игра может тормозить.

Решение: оставить такие логи только под `state.devMode`.

### Риск 2. Ссылки на имена без импорта

В некоторых местах кода нужно проверить, не используются ли имена, которые не импортированы и не объявлены.

Особенно проверить:

- `grid`;
- `towers`;
- `findPath`;
- `debugLogEvent`;
- `hideTowerInfo`.

Если ошибка `ReferenceError: ... is not defined`, это нужно исправлять импортом или обращением через `state`.

### Риск 3. Две реализации поиска пути

В коде есть логика поиска пути в `game.js` и импортируемый `findPath` из `gameActions.js`.

Нужно проверить, не дублируются ли алгоритмы и не расходятся ли правила движения.

### Риск 4. Диагонали и прямое движение

В одной части кода описан A* с диагоналями, а в `gameActions.js` соседние клетки указаны только в четырёх направлениях.

Это может давать разные результаты проверки пути и движения.

### Риск 5. Большие волны

Поздние волны могут создавать очень много врагов. Это может быть задумкой, но нужно тестировать производительность.

## 9. Как восстановить один файл из резервной ветки

Если сломан один файл, лучше восстановить только его, а не весь проект.

Пример:

```bash
git checkout backup-branch-name -- js/game.js
```

Потом проверить:

```bash
npm run dev
npm run build
```

И закоммитить восстановление:

```bash
git add js/game.js
git commit -m "restore: recover js/game.js from backup"
git push
```

## 10. Как полностью откатиться к резервной ветке

Делать только если проект сильно сломан.

Безопаснее сначала создать новую ветку от резерва:

```bash
git checkout backup-branch-name
git checkout -b restore-from-backup
```

Потом проверить проект.

Не нужно сразу force-push в main без понимания.

## 11. Как сравнить main и резерв

```bash
git diff backup-branch-name..main
```

Посмотреть список изменённых файлов:

```bash
git diff --name-only backup-branch-name..main
```

## 12. Как понять, что сломало проект

Использовать историю:

```bash
git log --oneline
```

Потом смотреть изменения конкретного коммита:

```bash
git show <commit_sha>
```

Если известно, что до определённого коммита работало, можно искать проблемный коммит.

## 13. Что фиксировать после восстановления

После любого восстановления обновить:

- `CHANGELOG.md`;
- `docs/BUGS_AND_FIXES.md`;
- при необходимости `docs/CURRENT_STATE.md`;
- при необходимости `docs/RECOVERY_LOG.md`.

## 14. Минимальная рабочая версия проекта

Минимально проект должен иметь:

- `index.html`;
- `package.json`;
- папку `js/`;
- `js/game.js`;
- `js/constants.js`;
- `js/state.js`;
- `js/gameActions.js`;
- `js/ui.js`;
- рабочую команду `npm run dev`.

## 15. Итог

Восстановление проекта должно идти по принципу:

```text
Сначала понять → потом проверить → потом восстановить → потом записать в документацию.
```

Документация является частью восстановления, потому что без неё снова будет непонятно, что произошло и почему.
