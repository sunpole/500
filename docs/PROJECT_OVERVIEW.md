# Обзор проекта 500

## Назначение проекта

`500` — браузерная игра в жанре tower defense, где игрок строит башни-животных и защищается от волн врагов-фруктов и овощей.

Проект начался как длинный единый HTML-файл, внутри которого находились HTML, CSS и JavaScript. После обсуждения и помощи со стороны знакомого разработчика проект был разделён на отдельные файлы и модули, а также получил сборку через Vite.

## Главная цель

Сделать понятный, расширяемый и аккуратно документированный игровой проект, который можно:

- запускать локально;
- развивать поэтапно;
- публиковать на GitHub Pages;
- безопасно изменять без потери старых решений;
- возвращать в работу из разных чатов и быстро понимать текущее состояние.

## Текущая идея игры

Игрок защищается от волн врагов. Враги идут по маршруту или игровой сетке, а игрок покупает и размещает башни. Башни атакуют врагов с разной скоростью, уроном, радиусом и типом атаки.

Визуальная идея проекта сейчас простая и дружелюбная:

- враги — фрукты и овощи;
- башни — животные;
- игра должна быть понятной, наглядной и пригодной для дальнейшего расширения.

2. Лазерная атака.
3. Ядовитая атака с периодическим уроном.

## Яд и периодический урон

Некоторые башни имеют блок `dot`.

Пример смысла параметров:

- `type` — тип эффекта, сейчас poison;
- `dps` — урон в секунду;
- `stackDuration` — длительность одного стака;
- `maxStacks` — максимальное количество стаков;
- `multiDps` — может ли урон складываться;
- `multiStacks` — могут ли стаки накапливаться.

Это важная механика, потому что она позволяет делать башни не только с прямым уроном, но и с накопительным эффектом.

## Волны

Волны хранятся в `waveData`. Каждая волна содержит список групп врагов.

Поля:

- `e` — индекс врага из `enemyData`;
- `n` — количество врагов;
- `d` — задержка или интервал появления.

Пример:

```js
{ enemies: [ { e: 0, n: 12, d: 0.2 } ] }
```

Это означает: выпустить 12 врагов типа `enemyData[0]` с указанным интервалом.

## Решение по Vite

Vite был добавлен как пример современного разделения проекта. Это значит, что проект теперь можно развивать не только как один HTML-файл, но и как нормальный фронтенд-проект с модулями.

Это полезно, потому что:

- легче разделять код;
- легче искать ошибки;
- легче выносить данные;
- легче публиковать готовую сборку;
- легче подключать будущие модули.

## Решение по JSON-данным

Обсуждалось вынесение базы волн из `js/constants.js` в отдельный JSON-файл.

Вывод: это можно делать без бэкенда. GitHub Pages умеет отдавать JSON-файлы как обычные статические файлы.

Будущая структура может быть такой:

```text
public/data/waves.json
public/data/enemies.json
public/data/towers.json
```

Загрузка:

```js
const response = await fetch('data/waves.json');
const waveData = await response.json();
```

Важно: JSON не поддерживает комментарии, поэтому комментарии нужно превращать в поля `note`, `wave`, `group`, `description`.

## Основной принцип развития

Проект должен развиваться осторожно:

1. Сначала документация.
2. Потом проверка текущей структуры.
3. Потом аккуратные малые патчи.
4. Потом вынос данных.
5. Потом балансировка.
6. Потом улучшение интерфейса.
7. Потом публикация и тестирование.

## Что не делать без отдельного решения

Не нужно резко переписывать игру целиком.

Не нужно ломать текущий рабочий вариант ради архитектуры.

Не нужно смешивать крупный рефакторинг и игровые изменения в одном коммите.

Не нужно удалять старые данные без резервной копии.

## Документационный стандарт

Каждая важная мысль по проекту должна фиксироваться в документации.

Каждый баг должен попадать в `docs/BUGS_AND_FIXES.md`.

Каждое изменение должно попадать в `CHANGELOG.md`.

Каждая будущая идея должна попадать в `ROADMAP.md` или `docs/DEVELOPMENT_LOG.md`.
