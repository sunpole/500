## 5. Что делать, если проект сломан

1. Открыть [Руководство по восстановлению](RESTORE_GUIDE.md).
2. Пройти [Диагностический чек-лист](DIAGNOSTIC_CHECKLIST.md).
3. Сверить ошибки с [Известными проблемами по аудиту кода](KNOWN_ISSUES_FROM_CODE_AUDIT.md).
4. Записать подтверждённый баг в [Баги и решения](BUGS_AND_FIXES.md).
5. Исправлять только один баг за раз.
6. После исправления обновить [Changelog](../CHANGELOG.md) и [Журнал восстановления](RECOVERY_LOG.md).

## 6. Что делать, если нужно понять код

1. Открыть [Текущее состояние](CURRENT_STATE.md).
2. Открыть [Карту модулей](MODULE_MAP.md).
3. Открыть [Инвентаризацию файлов](FILE_INVENTORY.md).
4. Смотреть код в таком порядке:

```text
index.html
js/state.js
js/constants.js
js/gameActions.js
js/ui.js
js/game.js
```

## 7. Что делать, если нужно менять баланс

1. Открыть [Структуру данных](DATA_STRUCTURE.md).
2. Проверить `enemyData`, `towerData`, `waveData`.
3. Не менять сразу врагов, башни и волны вместе.
4. После изменения проверить первые волны.
5. Записать изменение в [Changelog](../CHANGELOG.md).

## 8. Что делать, если нужно выносить данные в JSON

1. Прочитать [Структуру данных](DATA_STRUCTURE.md).
2. Прочитать [Vite](VITE_NOTES.md).
3. Прочитать [Архитектуру](ARCHITECTURE.md).
4. Сначала вынести только `waveData`.
5. Использовать путь:

```text
public/data/waves.json
```

6. Загружать как:

```js
fetch('data/waves.json')
```

7. Не забыть: JSON не поддерживает комментарии.

## 9. Главные известные подозрения по коду

Первичный аудит показал возможные проблемы:

- `grid` вместо `state.grid`;
- `towers` вместо `state.towers`;
- `money` вместо `state.money`;
- `placingTowerCell` вместо `state.placingTowerCell`;
- неимпортированный `findPath`;
- `conf.bbulletSpeed` вместо `conf.bulletSpeed`;
- `upgradeTower()` и `sellTower()` могут вызываться без координат;
- `getNextTowerType` и `getPrevTowerType` могут быть недоступны в `game.js`;
- слишком много `console.log` в игровом цикле;
- разные правила поиска пути.

Подробно: [Известные проблемы по аудиту кода](KNOWN_ISSUES_FROM_CODE_AUDIT.md).

## 10. Главные правила

- Не ломать `main` без резерва.
- Резервную ветку не трогать.
- Один патч — одна причина.
- Документация обновляется вместе с изменениями.
- Баг без записи считается потерянным багом.
- Исправление без проверки считается неподтверждённым.
