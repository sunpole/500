# Структура данных проекта 500

Этот документ описывает игровые данные: размеры поля, врагов, башни, волны и будущую идею переноса части данных в JSON.

## Где сейчас находятся данные

На момент создания документации основные игровые данные находятся в файле:

```text
js/constants.js
```

В нём находятся:

- `GRID_SIZE`;
- `CELL_SIZE`;
- `CANVAS_WIDTH`;
- `CANVAS_HEIGHT`;
- `enemyData`;
- `towerData`;
- `waveData`.

## Размеры игрового поля

```js
export const GRID_SIZE = 15;
export const CELL_SIZE = 40;
export const CANVAS_WIDTH = GRID_SIZE * CELL_SIZE;
export const CANVAS_HEIGHT = GRID_SIZE * CELL_SIZE;
```

Смысл:

- `GRID_SIZE` — количество клеток по ширине и высоте;
- `CELL_SIZE` — размер одной клетки в пикселях;
- `CANVAS_WIDTH` — ширина canvas;
- `CANVAS_HEIGHT` — высота canvas.

При текущих значениях поле имеет размер:

```text
15 * 40 = 600 пикселей
```

То есть canvas получается 600×600 пикселей.

## База врагов: enemyData

`enemyData` — массив объектов. Каждый объект описывает один тип врага.

Пример:

```js
{
  name: "Яблоко",
  hp: 80,
  speed: 250,
  color: '#ff3e3e',
  reward: 7,
  damage: 1
}
```

### Поля врага

| Поле | Смысл |
|---|---|
| `name` | Название врага |
| `hp` | Здоровье |
| `speed` | Скорость движения |
| `color` | Цвет для отрисовки |
| `reward` | Награда игроку за уничтожение |
| `damage` | Урон, если враг дошёл до цели |

## DOT / яд

Некоторые башни имеют поле `dot`.

Пример:

```js
{
  type: "poison",
  dps: 30,
  stackDuration: 10,
  maxStacks: 50,
  multiDps: true,
  multiStacks: true
}
```

### Поля DOT

| Поле | Смысл |
|---|---|
| `type` | Тип эффекта |
| `dps` | Урон в секунду |
| `stackDuration` | Длительность стака |
| `maxStacks` | Максимум стаков |
| `multiDps` | Складывается ли урон |
| `multiStacks` | Складываются ли стаки |

## База волн: waveData

`waveData` — массив волн. Каждая волна содержит массив групп врагов.

Пример:

```js
{
  enemies: [
    { e: 0, n: 12, d: 0.2 }
  ]
}
```

### Поля группы врагов

| Поле | Смысл |
|---|---|
| `e` | Индекс врага из `enemyData` |
| `n` | Количество врагов |
| `d` | Задержка/интервал выпуска |

## Почему индексы важны

Волны ссылаются на врагов по индексу.

Например:

```js
{ e: 0, n: 12, d: 0.2 }
```

`e: 0` означает первого врага в `enemyData`, то есть Яблоко.

Если поменять порядок врагов в `enemyData`, волны начнут ссылаться на других врагов. Поэтому порядок врагов нельзя менять без миграции волн.

## Потенциальная проблема текущей структуры

Сейчас волны ссылаются на врагов по числовым индексам. Это компактно, но рискованно.

Например, если удалить Банан или переставить врагов, баланс волн сломается.

Более безопасная будущая структура:

```json
{
  "enemy": "apple",
  "count": 12,
  "delay": 0.2
}
```

Для этого у врагов должны появиться стабильные `id`:

```json
{
  "id": "apple",
  "name": "Яблоко",
  "hp": 80
}
```

## Идея будущего JSON

Можно вынести данные в отдельные JSON-файлы.

Возможная структура:

```text
public/data/enemies.json
public/data/towers.json
public/data/waves.json
```

## Почему это можно без бэкенда

GitHub Pages отдаёт статические файлы. JSON-файл — это обычный статический файл.

Значит, браузер может загрузить его так:

```js
const response = await fetch('data/waves.json');
const data = await response.json();
```

Бэкенд для этого не нужен.

## Важное правило для Vite

Если файл лежит в `public/data/waves.json`, то обращаться к нему нужно так:

```js
fetch('data/waves.json')
```

Не так:

```js
fetch('public/data/waves.json')
```

Папка `public` становится корнем сайта при сборке.

## Почему не нужно сразу всё переносить

Перенос данных в JSON меняет способ запуска игры. Код должен дождаться загрузки данных перед стартом.

Поэтому правильный порядок:

1. Сначала документация.
2. Потом отдельный патч: вынести только `waveData`.
3. Проверить локально.
4. Проверить после `npm run build`.
5. Проверить на GitHub Pages.
6. Потом отдельный патч: вынести `enemyData`.
7. Потом отдельный патч: вынести `towerData`.

## Как сохранить комментарии волн при переносе

В JS сейчас можно писать:

```js
// Волна 5
```

В JSON так нельзя.

Варианты замены:

```json
{
  "wave": 5,
  "note": "Волна 5",
  "enemies": []
}
```

Или:

```json
{
  "id": "wave-005",
  "title": "Волна 5",
  "enemies": []
}
```

## Проверки после изменения данных

После любого изменения `enemyData`, `towerData` или `waveData` нужно проверить:

1. Игра запускается.
2. Первая волна стартует.
3. Все враги появляются.
4. Башни покупаются.
5. Башни атакуют.
6. Деньги начисляются.
7. Враги наносят урон при проходе.
8. Консоль браузера не показывает ошибок.
9. `npm run build` проходит успешно.
10. Собранная версия работает.

## Что считается багом в данных

Багами считаются:

- волна ссылается на несуществующий индекс врага;
- у врага нет обязательного поля;
- у башни нет обязательного поля;
- отрицательная цена;
- нулевой или отрицательный cooldown там, где код этого не ожидает;
- слишком большой `n`, из-за которого браузер зависает;
- JSON содержит комментарии;
- путь к JSON не работает на GitHub Pages;
- данные загружаются после старта игры, и игра падает.

## Рекомендация по балансу

Баланс лучше менять маленькими шагами.

Не стоит одновременно:

- менять здоровье врагов;
- менять цены башен;
- менять урон башен;
- менять количество врагов;
- менять скорость волн.

Иначе будет сложно понять, что именно сломало баланс.

## Рекомендованный порядок балансировки

1. Проверить первые 5 волн.
2. Проверить волны 10, 15, 20.
3. Проверить момент появления сильных врагов.
4. Проверить окупаемость дешёвых башен.
5. Проверить, не слишком ли сильны DOT-башни.
6. Проверить, не ломают ли лазеры баланс.
7. Проверить позднюю игру.
