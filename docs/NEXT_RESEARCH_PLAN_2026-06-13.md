# План следующего глубокого исследования репозитория 500

Дата: 2026-06-13.

Этот документ дополняет анализ поиска и нужен для следующего этапа работы, когда потребуется не только писать документацию, но и фактически проверять код.

## 1. Почему нужен отдельный план

GitHub code search для репозитория на момент проверки не индексирован. Поэтому поиск по ключевым словам через GitHub может не находить даже существующие строки.

Чтобы не делать ложных выводов, следующий аудит должен идти по прямому чтению файлов и локальной проверке.

## 2. Главные цели следующего аудита

1. Проверить фактическую структуру файлов.
2. Сопоставить структуру с `docs/FILE_INVENTORY.md`.
3. Найти реальные места, где объявлены данные игры.
4. Найти реальные места, где используются данные.
5. Проверить подозрения из `docs/KNOWN_ISSUES_FROM_CODE_AUDIT.md`.
6. Разделить неподтверждённые подозрения и подтверждённые баги.
7. Подготовить первый безопасный технический патч.

## 3. Что нужно проверить в первую очередь

### 3.1. Запуск

```bash
npm install
npm run dev
npm run build
```

Нужно записать:

- запускается ли проект;
- есть ли ошибки в терминале;
- есть ли ошибки в консоли браузера;
- создаётся ли папка `dist`;
- открывается ли игра после сборки.

### 3.2. Файлы

Проверить наличие:

```text
index.html
package.json
vite.config.js
js/constants.js
js/state.js
js/game.js
js/gameActions.js
js/ui.js
```

Если структура отличается, обновить `docs/FILE_INVENTORY.md` и `docs/MODULE_MAP.md`.

### 3.3. Данные

Найти:

```text
enemyData
towerData
waveData
```

Понять:

- где они объявлены;
- как экспортируются;
- где импортируются;
- используются ли числовые индексы;
- можно ли безопасно вынести сначала только `waveData`.

### 3.4. Подозрения по багам

Проверить:

```text
grid vs state.grid
towers vs state.towers
money vs state.money
placingTowerCell vs state.placingTowerCell
findPath import
conf.bbulletSpeed
upgradeTower/sellTower без координат
console.log в игровом цикле
```

Каждое подозрение после проверки должно получить статус:

- подтверждено;
- не подтверждено;
- требует дополнительной проверки;
- уже исправлено.

## 4. Формат фиксации результатов

Результаты проверки нужно записывать в:

```text
docs/DIAGNOSTIC_CHECKLIST.md
docs/BUGS_AND_FIXES.md
docs/RECOVERY_LOG.md
CHANGELOG.md
```

Если найден баг:

1. записать его в `BUGS_AND_FIXES.md`;
2. описать шаги воспроизведения;
3. описать ожидаемое поведение;
4. описать фактическое поведение;
5. только потом исправлять.

## 5. Первый безопасный технический патч

После диагностики первым кодовым патчем должен быть самый маленький подтверждённый фикс.

Не нужно начинать с JSON, если проект не запускается.

Приоритет:

1. исправить ошибку, которая мешает запуску;
2. исправить ошибку, которая ломает базовое действие игрока;
3. исправить очевидную опечатку;
4. удалить или ограничить лишние `console.log`, если они реально мешают;
5. только потом архитектура.

## 6. Когда можно переходить к JSON

К выносу `waveData` в JSON можно переходить только если:

- проект запускается;
- сборка работает;
- текущие данные найдены;
- понятно, где стартует игра;
- понятно, как дождаться загрузки JSON;
- есть план отката.

## 7. Итог

Следующий глубокий аудит должен быть не только документационным, но и техническим: чтение файлов, запуск, сборка, проверка консоли и фиксация подтверждённых багов.

Главное правило остаётся прежним: не ломать рабочее состояние и не смешивать разные типы изменений.
