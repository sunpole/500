# Полный аналитический отчёт по проекту 500 из текущего чата

Дата фиксации: 2026-06-13.

Этот документ дополняет уже существующую документацию и не заменяет её. Его задача — сохранить максимально широкий контекст из текущего чата: мысли, цели, решения, технологии, риски, слабые места, принципы разработки и возможные направления развития.

## 1. Главная цель проекта

Проект `500` рассматривается не просто как игра, а как долгосрочный учебно-практический проект, который должен быть понятен владельцу проекта и пригоден для аккуратного развития.

Главная цель:

- сохранить рабочий проект;
- не потерять идеи из чатов;
- документировать причины решений;
- сделать проект пригодным для возврата к любому этапу разработки;
- развивать игру маленькими безопасными шагами;
- иметь понятную дорожную карту;
- иметь возможность отката через резервные ветки, релизы и документацию.

Ключевая мысль: документация является частью проекта, а не второстепенным дополнением.

## 2. Смысл игры

`500` — браузерная игра в жанре tower defense.

Игрок защищает базу от волн врагов. Враги оформлены как фрукты и овощи. Башни оформлены как животные.

Такой образ делает игру:

- визуально простой;
- дружелюбной;
- понятной без тяжёлого фэнтези/военного сеттинга;
- удобной для расширения новыми врагами и башнями;
- подходящей для экспериментов с балансом.

## 3. Основные сущности

### 3.1. Игрок

Игрок управляет размещением башен, деньгами, защитой базы и развитием обороны.

Потенциальные параметры игрока:

- деньги;
- жизни базы;
- выбранный тип башни;
- состояние строительства;
- текущая волна;
- прогресс прохождения;
- статистика.

### 3.2. Враги

Враги — фрукты и овощи.

Уже зафиксированные образы:

- Яблоко;
- Банан;
- Авокадо;
- Арбуз;
- Дыня;
- Груша;
- Капуста;
- Свекла;
- Баклажан;
- Помидор;
- Тыква.

Смысл параметров врага:

- здоровье определяет, сколько урона нужно нанести;
- скорость определяет темп движения;
- награда определяет экономику;
- урон по базе определяет цену ошибки;
- цвет помогает визуально различать врагов.

### 3.3. Башни

Башни — животные.

Уже зафиксированные образы:

- Мышка;
- Геккон;
- Попугайчик;
- Совёнок;
- Лиса;
- Ёж;
- Орёл;
- Кенгуру;
- Рекс;
- Гепард;
- Дракон;
- Скорпион.

Смысл параметров башни:

- цена — экономическое ограничение;
- радиус — зона контроля;
- урон — сила атаки;
- перезарядка — частота атаки;
- скорость снаряда — ощущение динамики;
- тип атаки — роль башни;
- дополнительные эффекты — уникальность.

## 4. Игровые механики

### 4.1. Tower Defense основа

Базовая механика:

1. Враги появляются волнами.
2. Они движутся по маршруту или сетке.
3. Игрок покупает и размещает башни.
4. Башни автоматически атакуют врагов.
5. За уничтожение врагов игрок получает деньги.
6. Если враги доходят до базы, игрок теряет жизни.
7. Цель — выдержать волны.

### 4.2. Обычные снаряды

Обычный тип атаки подходит для базовых башен.

Важные параметры:

- цель;
- скорость снаряда;
- урон;
- попадание;
- удаление снаряда после попадания.

### 4.3. Лазерная атака

Лазер отличается от обычного снаряда тем, что может быть мгновенным или почти мгновенным.

Его можно использовать для башен с точечной атакой, визуальным лучом, быстрым уроном.

Риск: визуальные эффекты лазера могут усложнить код рендера, поэтому их лучше документировать и внедрять отдельно.

### 4.4. Яд / DOT

DOT — damage over time, урон со временем.

В проекте уже упоминались параметры:

- `type`;
- `dps`;
- `stackDuration`;
- `maxStacks`;
- `multiDps`;
- `multiStacks`.

Эта механика важна, потому что делает башни различными по роли:

- прямой урон;
- урон со временем;
- накопление эффекта;
- контроль сильных врагов;
- баланс против медленных толстых волн.

## 5. Волны

Волны хранятся как данные, а не как отдельная логика.

Текущий смысл структуры:

- `e` — индекс врага;
- `n` — количество;
- `d` — задержка/интервал появления.

Пример смысла:

```js
{ enemies: [ { e: 0, n: 12, d: 0.2 } ] }
```

Это означает: выпустить 12 врагов типа `enemyData[0]` с интервалом `0.2`.

Важное решение: волны в будущем можно вынести в JSON, но только отдельным патчем и после проверки текущей сборки.

## 6. Технологии

### 6.1. Текущий стек

Проект использует:

- HTML;
- CSS;
- JavaScript;
- JavaScript modules;
- npm;
- Vite;
- GitHub;
- GitHub Pages в перспективе.

### 6.2. Почему Vite

Vite появился после PR от знакомого разработчика.

Смысл Vite:

- запуск локального сервера;
- работа с модулями;
- удобная разработка;
- сборка в `dist`;
- подготовка к GitHub Pages.

Основные команды:

```bash
npm install
npm run dev
npm run build
```

### 6.3. Почему не спешить с усложнением

Хотя Vite полезен, проект должен оставаться понятным владельцу.

Нельзя вводить сложные архитектурные решения только ради красоты. Любое усложнение должно отвечать на практическую проблему:

- легче искать ошибки;
- легче менять баланс;
- легче добавлять врагов;
- легче публиковать;
- легче тестировать.

## 7. Решение по JSON

Обсуждалась идея вынести данные из `js/constants.js`.

Возможная будущая структура:

```text
public/data/enemies.json
public/data/towers.json
public/data/waves.json
```

Почему это полезно:

- баланс можно менять без копания в логике;
- данные становятся ближе к таблице;
- проще сравнивать версии;
- проще документировать волны;
- можно сделать редактор данных в будущем.

Ограничение:

JSON не поддерживает комментарии.

Поэтому вместо комментариев нужно использовать поля:

```json
{
  "wave": 5,
  "note": "Первая волна с Авокадо",
  "enemies": []
}
```

## 8. Главные риски проекта

### 8.1. Потеря истории

Часть решений была в чатах. Если их не фиксировать, будущая разработка станет хаотичной.

Меры защиты:

- вести `docs/CHAT_CONTEXT.md`;
- вести `docs/DEVELOPMENT_LOG.md`;
- вести `CHANGELOG.md`;
- добавлять отдельные аналитические документы;
- не удалять старые документы без причины.

### 8.2. Поломка рабочего состояния

Риск: крупный рефакторинг может сломать игру.

Меры защиты:

- резервная ветка перед крупными изменениями;
- маленькие коммиты;
- не смешивать разные типы изменений;
- сначала документация и диагностика;
- потом код.

### 8.3. Смешивание баланса и архитектуры

Баланс и архитектура должны меняться отдельно.

Плохой патч:

- одновременно выносит данные в JSON;
- меняет урон башен;
- меняет скорость врагов;
- меняет интерфейс.

Хороший патч:

- только выносит `waveData` в JSON;
- или только чинит баг;
- или только добавляет документ;
- или только меняет баланс одной группы.

### 8.4. Непонятность для владельца

Проект должен быть понятен Антону, а не только опытному разработчику.

Поэтому документация должна объяснять:

- что такое PR;
- что такое Vite;
- что такое сборка;
- где лежат данные;
- как запускать;
- как безопасно менять.

## 9. Принятая стратегия разработки

1. Сначала читать документацию.
2. Проверять текущее состояние проекта.
3. Не трогать резервные ветки.
4. Делать изменения в `main` только осознанно.
5. Код менять маленькими патчами.
6. Все изменения фиксировать в документации.
7. Все найденные баги записывать.
8. Не удалять старый контекст.
9. После каждого важного этапа обновлять roadmap.
