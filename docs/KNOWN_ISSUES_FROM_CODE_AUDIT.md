# Известные проблемы по первичному аудиту кода

Дата первичного аудита: 2026-06-13.

Этот документ фиксирует подозрительные места, найденные при чтении кода. Это не означает, что все пункты уже подтверждены запуском. Но каждый пункт нужно проверить в браузере и при необходимости перенести в `docs/BUGS_AND_FIXES.md` как подтверждённый баг.

## Важно

На этом этапе код не исправлялся. Документ только фиксирует возможные причины поломок.

## 1. Возможная ошибка `grid is not defined`

### Где замечено

`js/game.js`, обработчик клика по полю.

Подозрительное обращение:

```js
grid[y][x].tower
```

### Почему это риск

В проекте состояние сетки хранится в:

```js
state.grid
```

Если переменная `grid` нигде не объявлена глобально, при клике может быть ошибка:

```text
ReferenceError: grid is not defined
```

### Предполагаемое исправление

Заменить:

```js
grid[y][x]
```

на:

```js
state.grid[y][x]
```

### Статус

Требует проверки запуском.

## 3. Возможная ошибка `findPath is not defined` в game.js

### Где замечено

`js/game.js`, функция `updateEnemies()`.

Подозрительное обращение:

```js
const newPath = findPath([cx, cy], [GRID_SIZE - 1, GRID_SIZE - 1]);
```

### Почему это риск

`findPath()` экспортируется из `gameActions.js`, но в импорте `game.js` она не указана.

Текущий импорт:

```js
import { Bullet, placeTower, sellTower, spawnEnemy, getDeltaTime, distance, applyDotEffect, generateEnemyPath } from './gameActions.js';
```

### Предполагаемое исправление

Добавить `findPath` в импорт:

```js
import { Bullet, placeTower, sellTower, spawnEnemy, getDeltaTime, distance, applyDotEffect, generateEnemyPath, findPath } from './gameActions.js';
```

Или использовать единую функцию поиска пути внутри одного модуля.

### Статус

Требует проверки запуском.

## 4. Возможная проблема: разные алгоритмы пути

### Где замечено

В `game.js` есть `hasEnemyPath()` с 8 направлениями и проверкой диагоналей.

В `gameActions.js` есть `findPath()` с 4 направлениями.

### Почему это риск

Проверка возможности строительства может разрешать путь по одним правилам, а реальное движение врагов может использовать другие правила.

Это может привести к странным ситуациям:

- башню разрешили поставить, но враг не может пройти;
- башню запретили поставить, хотя путь есть;
- враги перестраивают путь не так, как ожидает проверка.

### Предполагаемое исправление

Оставить один единый алгоритм пути и использовать его везде.

### Статус

Архитектурный риск. Исправлять отдельным патчем после проверки.

## 8. Возможная путаница между UI-функциями и глобальными функциями

### Где замечено

`ui.js` создаёт HTML-кнопки с inline `onclick`:

```html
onclick="selectTowerType(...)"
onclick="showTowerInfo(...)"
onclick="upgradeTower()"
onclick="sellTower()"
onclick="clearTowerSelection()"
```

В `game.js` в конце функции частично экспортируются в `window`:

```js
window.selectTowerType = ...
window.clearTowerSelection = ...
window.upgradeTower = ...
window.downgradeTower = ...
window.showTowerInfo = showTowerInfo
window.hideTowerInfo = hideTowerInfo
window.sellTower = sellTower
window.restartGame = restartGame
```

### Почему это риск

В модульном JavaScript функции не становятся глобальными автоматически.

Если кнопка использует inline `onclick`, функция должна быть доступна в `window`.

Некоторые функции существуют в `ui.js`, но не экспортируются в глобал напрямую из него.

### Статус

Частично решено в `game.js`, но нужно проверить кнопки в браузере.

## 9. Возможная ошибка в глобальных `upgradeTower` и `downgradeTower`

### Где замечено

В конце `js/game.js`:

```js
window.upgradeTower = (x, y) => {
  let nextType = getNextTowerType(curType);
}

window.downgradeTower = (x, y) => {
  let prevType = getPrevTowerType(curType);
}
```

### Почему это риск

`getNextTowerType()` и `getPrevTowerType()` определены в `ui.js`, но не экспортируются и не импортируются в `game.js`.

Если нажать upgrade/downgrade, может быть:

```text
ReferenceError: getNextTowerType is not defined
ReferenceError: getPrevTowerType is not defined
```

### Предполагаемое исправление

Варианты:

1. Экспортировать функции из `ui.js` и импортировать в `game.js`.
2. Перенести upgrade/downgrade в один модуль.
3. Убрать дублирование и сделать единый источник логики.

### Статус

Требует проверки запуском.

## 10. Кнопки `upgradeTower()` и `sellTower()` без координат

### Где замечено

`js/ui.js`, `createUIButtons()`:

```js
html += `<button onclick="upgradeTower()">Апгрейд</button>`;
html += `<button onclick="sellTower()">Продать</button>`;
```

### Почему это риск

Глобальная `upgradeTower` в `game.js` ожидает координаты:

```js
window.upgradeTower = (x, y) => { ... }
```

`sellTower` из `gameActions.js` тоже ожидает координаты:

```js
sellTower(x, y)
```

Если вызвать без координат, может быть ошибка или некорректное поведение.

### Возможное решение

Либо убрать эти кнопки из общей панели, либо сделать их активными только при выбранной конкретной башне и передавать координаты.

### Статус

Вероятный UI-баг.

## 11. Слишком много console.log в игровом цикле

### Где замечено

`js/game.js`:


### Почему это риск

Обернуть в:


или удалить после отладки.

## 13. Возможная проблема победы

### Где замечено

`js/game.js`, `checkVictoryCondition()`:

```js
state.wave > waveData.length
```

### Почему это риск

Если `state.wave` увеличивается до `waveData.length`, условие `>` может не сработать в ожидаемый момент.

Возможно нужно:

```js
state.wave >= waveData.length
```

Но это нужно проверять по фактической логике запуска волн.

### Статус

Требует тестирования.
