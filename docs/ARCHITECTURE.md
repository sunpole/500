# Архитектура проекта 500

Этот документ описывает текущую и желаемую архитектуру проекта.

## История архитектуры

Проект начинался как один длинный файл:

```text
HTML + CSS + JS в одном файле
```

Такой подход удобен для быстрого старта, но становится сложным при росте проекта.

После просмотра проекта Даник Египцев подготовил PR, где:

- разделил код на файлы;
- разделил JavaScript на модули;
- добавил Vite;
- сделал запуск через `npm run dev`.

## Текущий подход

Проект теперь развивается как модульный фронтенд-проект.

Основные части:

- HTML — структура страницы;
- CSS — внешний вид;
- JS — логика игры;
- constants/data — настройки, враги, башни, волны;
- Vite — запуск и сборка.

## Главный архитектурный принцип

Код, данные и документация должны быть разделены.

Идеально:

```text
Код отвечает за поведение.
Данные отвечают за баланс и содержимое.
Документация отвечает за смысл и историю решений.
```

## Что сейчас находится в constants.js

Файл `js/constants.js` сейчас содержит:

- размеры игрового поля;
- базу врагов;
- базу башен;
- базу волн.

Это рабочий вариант, но в будущем его можно улучшить.

## Почему constants.js может стать слишком большим

Если в одном файле хранить всё, возникают проблемы:

- трудно искать нужную волну;
- трудно редактировать баланс;
- легко случайно сломать синтаксис;
- данные смешиваются с техническими константами;
- сложнее подключать редактор уровней или волн.

## Желаемая будущая архитектура данных

```text
js/constants.js
public/data/enemies.json
public/data/towers.json
public/data/waves.json
```

Где:

- `constants.js` — только технические параметры;
- `enemies.json` — база врагов;
- `towers.json` — база башен;
- `waves.json` — база волн.

## Возможная архитектура JS-модулей

js/

## Назначение будущих модулей

### main.js

Точка входа. Запускает загрузку данных и старт игры.

### constants.js

Хранит технические константы, которые не являются балансной базой.

### gameState.js

Хранит состояние игры:

- деньги;
- жизни;
- текущая волна;
- список врагов;
- список башен;
- список снарядов;
- статус игры.

### renderer.js

Отвечает за отрисовку canvas:

- поле;
- башни;
- враги;
- снаряды;
- эффекты;
- интерфейсные подсказки.

### input.js

Отвечает за ввод игрока:

- клик по клетке;
- выбор башни;
- установка башни;
- наведение мыши;
- возможно, сенсорное управление.

### enemies.js

Логика врагов:

- создание врага;
- движение;

- смерть;

### towers.js

Логика башен:

- выбор цели;

### bullets.js

Логика снарядов:

- создание;

- удаление.

### effects.js

Логика эффектов:

- яд;
- стаки;
- лазер;

### waves.js

Логика волн:

### dataLoader.js

Загрузка данных из JSON:

- враги;

### balance.js

Будущий модуль для расчётов баланса:

- прочность волн;
- рекомендации по балансу.

## Важное правило рефакторинга

Нельзя делать весь рефакторинг сразу.

Правильный порядок:

1. Документация.
2. Проверка текущего запуска.
3. Малое разделение одного модуля.
4. Проверка.
5. Коммит.
6. Запись в CHANGELOG.
7. Следующий модуль.

## Что нельзя смешивать

Не смешивать в одном патче:

- перенос данных в JSON;
- изменение баланса;
- изменение UI;
- изменение логики волн;
- изменение публикации GitHub Pages.

Иначе будет сложно найти причину ошибки.

## Архитектура загрузки JSON

Если данные вынесены в JSON, игра должна стартовать после загрузки данных.

Пример:

```js
async function init() {
  const enemies = await loadJson('data/enemies.json');
  const towers = await loadJson('data/towers.json');
  const waves = await loadJson('data/waves.json');

  startGame({ enemies, towers, waves });
}

init();
```

## Обработка ошибок загрузки

Если JSON не загрузился, игрок должен увидеть понятную ошибку, а не пустой экран.

Пример будущего подхода:

```js
try {
  await init();
} catch (error) {
  console.error(error);
  showError('Не удалось загрузить данные игры. Проверьте файлы data/*.json');
}
```

## Валидация данных

В будущем желательно добавить проверку данных.

Проверять:

- у каждого врага есть `id`;
- `id` уникальны;
- у каждой башни есть `id`;
- волны ссылаются только на существующих врагов;
- количество врагов положительное;
- задержки не отрицательные;
- цена башен положительная;
- cooldown больше нуля.

## Возможный файл валидатора

```text
js/dataValidator.js
```

Он может проверять JSON при старте и выводить ошибки в консоль.

## Почему это важно

Когда база волн станет большой, ошибки будут неизбежны.

Автоматическая валидация поможет находить:

- опечатки в id;
- пропущенные поля;
- неправильные числа;
- волны с несуществующими врагами.

## Архитектура GitHub Pages

Так как проект Vite, есть два уровня:

1. Исходники в репозитории.
2. Готовая сборка `dist`.

Для публикации нужно понять, как именно настроен GitHub Pages.

Возможные варианты:

- GitHub Pages берёт файлы из `main`;
- GitHub Pages берёт файлы из папки `/docs`;
- GitHub Pages публикует сборку через GitHub Actions;
- GitHub Pages требует отдельную ветку для `dist`.

Это нужно проверить отдельно.

## Риски архитектуры

### Риск 1: пути

На GitHub Pages проект может быть в `/500/`, поэтому абсолютные пути могут ломаться.

### Риск 2: асинхронность

JSON загружается не мгновенно. Игра должна ждать загрузки.

### Риск 3: индексы

Волны сейчас используют числовые индексы врагов.

### Риск 4: большой рефакторинг

Если переписать всё сразу, сложно будет откатиться.

## Архитектурная цель

Итоговая архитектура должна быть такой, чтобы:

- Антон понимал структуру;
- новый помощник мог быстро войти в проект;
- данные можно было редактировать отдельно;
- ошибки было легче искать;
- игра могла расти;
- GitHub Pages работал стабильно.

## Текущий вывод

Текущая архитектура уже сделала важный шаг от одного файла к модульному проекту.

Следующий правильный шаг — не переписывать игру, а стабилизировать документацию, проверить запуск и только потом постепенно выносить данные и улучшать структуру.
