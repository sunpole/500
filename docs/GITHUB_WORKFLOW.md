# GitHub-процесс для проекта 500

Этот документ описывает, как безопасно работать с репозиторием `sunpole/500`.

Цель — не потерять рабочий код, не сломать основную ветку и всегда иметь возможность откатиться.

## Главные правила

1. Основная рабочая ветка — `main`.
2. Перед крупными изменениями делается резервная ветка.
3. Перед крупными изменениями желательно делать release/tag.
4. Резервную ветку не трогать.
5. Каждый смысловой блок изменений — отдельный коммит.
6. Документацию обновлять вместе с изменениями.
7. Не смешивать в одном коммите разные типы изменений.

## Что уже сделано вручную

Перед началом наполнения документации владелец проекта вручную сделал:

- релиз/точку отката;
- резервную ветку.

Поэтому дальнейшая документация добавляется в `main`.

## Что такое ветка

Ветка — это отдельная линия разработки.

Пример:

- `main` — основная версия;
- `backup-before-docs` — резерв перед документацией;
- `feature-json-waves` — будущая ветка для выноса волн в JSON.

## Что такое релиз или тег

Релиз/тег — это зафиксированная точка состояния проекта.

Он нужен, чтобы можно было сказать:

> Вот состояние проекта до большого изменения.

Если что-то сломалось, можно вернуться к этой точке.

## Что такое Pull Request

Pull Request — предложение изменений.

Он нужен, когда:

- другой человек предлагает изменения;
- нужно посмотреть разницу перед принятием;
- нужно обсудить код;
- нельзя сразу писать в `main`.

## Рекомендуемый процесс перед крупной задачей

```bash
git checkout main
git pull origin main

git branch backup-before-task-name
git push origin backup-before-task-name

git tag pre-task-name
git push origin pre-task-name
```

После этого можно работать в `main` или в отдельной feature-ветке.

## Более безопасный процесс через отдельную ветку

Для больших изменений лучше так:

```bash
git checkout main
git pull origin main

git checkout -b feature-json-waves
```

После изменений:

```bash
git add .
git commit -m "feat: move waves data to json"
git push origin feature-json-waves
```

Затем создать Pull Request на GitHub.

## Когда можно работать прямо в main

Прямо в `main` можно работать, если:

- это только документация;
- это маленькое безопасное исправление;
- уже есть свежая резервная ветка и релиз;
- изменение не ломает код игры;
- понятно, что именно меняется.

## Когда нельзя работать прямо в main

Лучше не работать прямо в `main`, если:

- меняется архитектура;
- переносится база данных;
- переписывается логика игры;
- меняется система волн;
- меняется сборка;
- меняется публикация GitHub Pages;
- есть риск, что игра перестанет запускаться.

## Стандарт коммитов

Рекомендуемый стиль сообщений:

```text
docs: add project overview
feat: move waves to json
fix: correct wave loading path
refactor: split tower logic
balance: adjust early waves
chore: update vite config
```

Типы:

- `docs` — документация;
- `feat` — новая функция;
- `fix` — исправление бага;
- `refactor` — изменение структуры без изменения поведения;
- `balance` — правка игрового баланса;
- `chore` — технические служебные изменения.

## Что записывать в CHANGELOG.md

В `CHANGELOG.md` нужно писать:

- дату;
- что изменено;
- почему изменено;
- какие файлы затронуты;
- есть ли риск;
- как проверить.

## Что записывать в BUGS_AND_FIXES.md

В `docs/BUGS_AND_FIXES.md` нужно писать:

- описание бага;
- где обнаружен;
- как воспроизвести;
- причина;
- решение;
- статус;
- дата;
- связанные файлы.

## Что записывать в DEVELOPMENT_LOG.md

В `docs/DEVELOPMENT_LOG.md` нужно писать:

- идеи;
- рассуждения;
- решения;
- почему выбрали именно такой путь;
- что отложили;
- что отклонили;
- что нужно обсудить позже.

## Работа с GitHub Pages

Если проект использует Vite, обычно готовая версия собирается командой:

```bash
npm run build
```

Результат появляется в папке:

```text
dist/
```

Для публикации на GitHub Pages нужно отдельно проверить настройки Pages в репозитории.

Варианты публикации:

1. Публиковать из ветки и папки, если GitHub Pages настроен на статические файлы.
2. Публиковать через GitHub Actions.
3. Публиковать содержимое `dist` специальным способом.

Важно: исходники Vite и готовая сборка — не одно и то же.

## Локальная проверка проекта

После клонирования:

```bash
npm install
npm run dev
```

После этого Vite покажет локальный адрес, например:

```text
http://localhost:5173/
```

Для проверки сборки:

```bash
npm run build
```

Если сборка прошла успешно, появится папка `dist`.

## Проверка перед коммитом

Перед коммитом желательно проверить:

```bash
npm run build
```

Также открыть проект локально и проверить:

- страница открывается;
- нет ошибок в консоли;
- игра стартует;
- башни ставятся;
- волны идут;
- враги получают урон;
- деньги начисляются;
- игра не зависает.

## Запрещённые опасные действия без резерва

Нельзя без резерва:

- удалять файлы;
- переписывать `constants.js`;
- менять формат `waveData`;
- менять импорты;
- менять конфиг Vite;
- менять публикацию Pages;
- принимать большой PR без просмотра `Files changed`.

## Как принимать PR

Перед принятием PR:

1. Открыть PR.
2. Прочитать описание.
3. Открыть вкладку `Files changed`.
4. Посмотреть, какие файлы изменены.
5. Проверить, нет ли удаления важного кода.
6. Если возможно, запустить локально.
7. Если всё хорошо — нажать `Merge pull request`.
8. После merge проверить `main`.

## Как откатиться вручную

Если что-то сломалось после коммита, можно:

1. Вернуться к резервной ветке.
2. Вернуться к тегу/релизу.
3. Создать новый коммит, который возвращает старое состояние.
4. Не удалять историю без понимания последствий.

## Что делать, если непонятно

Если непонятно, что делает изменение:

- не принимать его сразу;
- попросить объяснение;
- сравнить файлы;
- проверить локально;
- записать вопрос в `DEVELOPMENT_LOG.md`.

## Специальное правило проекта 500

Так как Антон учится GitHub и совместной работе, документация должна объяснять не только итог, но и смысл действий.

Например, не просто:

```text
Запусти npm run build.
```

А так:

```text
Команда npm run build создаёт готовую версию сайта в папке dist. Эту папку можно использовать для публикации.
```

## Итог

GitHub-процесс проекта должен быть осторожным:

- сначала резерв;
- потом маленькие изменения;
- потом проверка;
- потом документация;
- потом коммит;
- потом публикация.

Главная цель — не скорость, а сохранность проекта и понятная история развития.
