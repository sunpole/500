# Протокол патчей проекта 500

Этот документ описывает, как правильно вносить изменения в проект.

Цель: каждый патч должен быть понятен, проверяем и обратим.

## Что такое патч

Патч — это одно осмысленное изменение проекта.

Примеры хороших патчей:

- исправить `grid` на `state.grid`;
- исправить опечатку `bbulletSpeed`;
- убрать лишний `console.log` из игрового цикла;
- вынести `waveData` в JSON;
- добавить проверку данных волн;
- обновить README.

Примеры плохих патчей:

- переписать всю игру;
- одновременно менять баланс, UI, архитектуру и GitHub Pages;
- удалить старый код без объяснения;
- исправить баг и сразу поменять стиль игры;
- сделать много разных правок без описания.

## Главный принцип

```text
Один патч = одна причина изменения.
```

## Перед каждым патчем

Проверить:

1. Есть ли свежий резерв.
2. Понятно ли, что меняется.
3. Понятно ли, как проверить.
4. Нужно ли обновить документацию.
5. Не смешиваются ли разные типы изменений.

## Типы патчей

### docs

Только документация.

Пример:

```text
docs: add restore guide
```

### fix

Исправление бага.

Пример:

```text
fix: use state.grid in click handler
```

### feat

Новая функция.

Пример:

```text
feat: load waves from json
```

### refactor

Изменение структуры без изменения поведения.

Пример:

```text
refactor: move pathfinding to gameActions
```

### balance

Правки баланса.

Пример:

```text
balance: reduce late wave enemy count
```

### chore

Служебные изменения.

Пример:

```text
chore: add vite config for github pages
```

## Пример патча исправления

```text
Название:
fix: use state.money in tower info

Тип:
fix

Причина:
В ui.js используется money без state, что может вызвать ReferenceError.

Что изменено:
money заменено на state.money.

Файлы:
js/ui.js
docs/BUGS_AND_FIXES.md
CHANGELOG.md

Как проверить:
1. npm run dev
2. открыть информацию о башне
3. убедиться, что ошибка money is not defined не появляется

Риски:
Низкие.

Как откатить:
Вернуть строку из предыдущего коммита.
```

## Порядок после исправления бага

1. Исправить код.
2. Проверить локально.
3. Обновить `docs/BUGS_AND_FIXES.md`.
4. Обновить `CHANGELOG.md`.
5. При необходимости обновить `docs/CURRENT_STATE.md`.
6. Сделать коммит.

## Порядок после архитектурного изменения

1. Создать резервную ветку или feature-ветку.
2. Сделать маленькое изменение.
3. Проверить запуск.
4. Проверить сборку.
5. Обновить `docs/ARCHITECTURE.md`.
6. Обновить `CHANGELOG.md`.
7. Обновить `docs/FILE_INVENTORY.md`, если появились новые файлы.

## Порядок после изменения данных

Если меняются `enemyData`, `towerData`, `waveData` или JSON:

1. Проверить синтаксис.
2. Проверить запуск.
3. Проверить первые волны.
4. Проверить, что индексы врагов не сломались.
5. Обновить `docs/DATA_STRUCTURE.md`.
6. Обновить `CHANGELOG.md`.
7. Если это баланс — обновить roadmap/заметки баланса.

## Порядок после изменения UI

1. Проверить кнопки.
2. Проверить inline onclick.
3. Проверить глобальные функции `window.*`.
4. Проверить мобильный вид.
5. Проверить консоль.
6. Обновить документацию.

## Минимальная проверка после любого патча

```bash
npm run dev
npm run build
```

В браузере:

- нет ошибок в консоли;
- игра открывается;
- canvas виден;
- UI виден.

## Расширенная проверка игрового патча

- старт первой волны;
- установка башни;
- стрельба башни;
- смерть врага;
- начисление денег;
- переход к следующей волне;
- отсутствие зависания.

## Как откатывать плохой патч

Если патч маленький, лучше сделать обратный коммит.

Если патч большой и всё сломал, сравнить с резервной веткой или релизом.

Нельзя делать `force push` без полного понимания последствий.

## Что запрещено

Запрещено:

- исправлять несколько независимых багов одним коммитом, если можно разделить;
- менять баланс вместе с архитектурой;
- удалять документацию;
- менять резервную ветку;
- принимать большой PR без чтения diff;
- оставлять баг без записи.

## Итог

Протокол патчей нужен, чтобы проект можно было восстановить не только по коду, но и по истории решений.
