# Vite в проекте 500

Этот документ объясняет, зачем в проекте появился Vite и как с ним работать.

## Почему появился Vite

Проект изначально был длинным файлом HTML + CSS + JS. Это удобно для старта, но со временем становится трудно:

- искать код;
- разделять логику;
- поддерживать стили;
- хранить данные;
- подключать модули;
- публиковать проект аккуратно.

Даник Египцев подготовил PR, где разделил проект на файлы и модули и добавил Vite.

## Что такое Vite простыми словами

Vite — это инструмент, который помогает запускать и собирать фронтенд-проект.

Он делает две главные вещи:

1. Запускает проект локально во время разработки.
2. Собирает готовую версию сайта для публикации.

## Текущие команды проекта

В `package.json` есть:

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  }
}
```

Это даёт команды:

```bash
npm run dev
npm run build
```

## Установка зависимостей

После скачивания проекта нужно выполнить:

```bash
npm install
```

Эта команда создаёт папку `node_modules` и устанавливает Vite.

## Локальный запуск

```bash
npm run dev
```

После запуска Vite покажет адрес, обычно похожий на:

```text
http://localhost:5173/
```

Этот адрес нужно открыть в браузере.

## Сборка проекта

```bash
npm run build
```

После сборки появляется папка:

```text
dist/
```

В ней находится готовый сайт.

## Важное отличие исходников от сборки

Исходники — это рабочие файлы проекта:

```text
index.html
js/
css/
package.json
```

Сборка — это готовый результат для публикации:

```text
dist/
```

На GitHub Pages часто публикуется именно готовая сборка, а не исходные файлы.

## Папка public

В Vite есть особая папка:

```text
public/
```

Всё, что лежит в `public`, копируется в корень готового сайта.

Например:

```text
public/data/waves.json
```

После сборки будет доступно как:

```text
data/waves.json
```

Поэтому в коде нужно писать:

```js
fetch('data/waves.json')
```

А не:

```js
fetch('public/data/waves.json')
```

## Почему это важно для JSON

Если данные волн будут вынесены в JSON, лучше положить файл сюда:

```text
public/data/waves.json
```

Тогда он будет работать:

- локально через `npm run dev`;
- после `npm run build`;
- на GitHub Pages.

## Возможная проблема путей на GitHub Pages

Если сайт опубликован не в корне домена, а в подпапке, например:

```text
https://sunpole.github.io/500/
```

нужно внимательно проверять пути к файлам.

Относительный путь:

```js
fetch('data/waves.json')
```

обычно безопаснее, чем абсолютный:

```js
fetch('/data/waves.json')
```

Потому что абсолютный путь может искать файл от корня домена, а не от проекта `/500/`.

## Что нельзя делать без проверки

Не стоит без проверки менять пути:

```js
import ... from ...
fetch(...)
```

Пути — частая причина, почему локально всё работает, а на GitHub Pages нет.

## Как проверить Vite-проект

1. Установить зависимости:

```bash
npm install
```

2. Запустить локально:

```bash
npm run dev
```

3. Открыть адрес Vite.

4. Проверить игру.

5. Собрать:

```bash
npm run build
```

6. Убедиться, что ошибок нет.

## Что писать в документацию при изменении Vite

Если меняется Vite, нужно обновить:

- `README.md`;
- `docs/VITE_NOTES.md`;
- `docs/GITHUB_WORKFLOW.md`;
- `CHANGELOG.md`.

## Возможные будущие задачи по Vite

- Проверить настройки GitHub Pages.
- Добавить инструкцию публикации именно для этого репозитория.
- Добавить `preview` script:

```json
"preview": "vite preview"
```

- Проверить, нужен ли `base` в `vite.config.js` для GitHub Pages.

## Возможный vite.config.js для GitHub Pages

Если проект публикуется по адресу:

```text
https://sunpole.github.io/500/
```

может понадобиться:

```js
import { defineConfig } from 'vite';

export default defineConfig({
  base: '/500/'
});
```

Но это нужно добавлять только после проверки текущей публикации.

## Почему не нужно спешить с конфигом

Если сейчас сайт уже работает, не нужно менять `vite.config.js` без необходимости.

Сначала нужно проверить:

- как настроен GitHub Pages;
- откуда он публикует сайт;
- работает ли текущая сборка;
- есть ли ошибки путей.

## Итог

Vite — полезный шаг для проекта. Он делает проект более профессиональным и удобным, но требует аккуратности с путями, сборкой и публикацией.

Главное правило: любые изменения в Vite и путях проверять локально и после сборки.
